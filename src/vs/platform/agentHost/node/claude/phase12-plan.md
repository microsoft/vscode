# Phase 12 — Subagents

> Last updated: 2026-05-14 after live-SDK probing + 4-model council review (GPT-5.5, Claude Opus 4.6, GPT-5.3-Codex, plus original 3-model on 2026-05-13).

**Status:** ✅ **DONE** (production landed + E2E validated 2026-05-15)

## Goal

Make Claude subagents (sessions spawned by the SDK's `Task`/`Agent` tool) first-class for clients. The host detects subagent starts in the live SDK stream, routes inner messages to a child session URI, completes that child session, and serves transcripts on demand for both live and restored sessions — closing the [Phase 13](./phase13-plan.md) deferral noted at [`claudeAgent.ts:610`](./claudeAgent.ts#L610).

## Scope

**In scope**

- **`IClaudeTranscriptService.getTranscript(uri)` — single ingress** for any transcript fetch (parent or subagent URI). Branches once internally on `isSubagentSession(uri)`; consumers (`ClaudeAgent.getSessionMessages`) become one-liners. Hides every URI-shape branch from the rest of the codebase. See [CONTEXT.md M7 §"Subagent correlation"](./CONTEXT.md).
- **`IClaudeSubagentResolver` — subagent strategy chain** behind `getTranscript`. Owns all `subagentURI → Turn[]` complexity. Two opportunistic feeders (`noteLiveAgentId`, `notePrimingFromParent`) populate an in-memory cache; chain falls through to four named strategies on miss.
- Live detection: emit [`IAgentSubagentStartedSignal`](../../common/agentService.ts#L350) the first time the SDK reports a previously-unseen `parent_tool_use_id` on an inner message, sourcing `agentName` / `agentDisplayName` / `agentDescription` from the originating `Task`/`Agent` `tool_use` block.
- Inner-message routing: every [`AgentSignal`](../../common/agentService.ts#L286) derived from a live SDK message whose envelope `parent_tool_use_id` is non-null carries `parentToolCallId` so [`AgentSideEffects`](../agentSideEffects.ts) routes it to the subagent session.
- **`pending_confirmation` routing for inner subagent tools** — thread the SDK's `CanUseTool.agentID` ([`sdk.d.ts:187`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L187)) through [`claudeCanUseTool.ts`](./claudeCanUseTool.ts) and [`claudeAgentSession.ts`](./claudeAgentSession.ts) so `pending_confirmation` signals land with `parentToolCallId` set when the tool runs inside a subagent.
- Live completion: emit [`IAgentSubagentCompletedSignal`](../../common/agentService.ts#L378) when (a) the parent `tool_use`'s `tool_result` arrives **for foreground subagents only**, or (b) `SDKTaskNotificationMessage` with terminal status arrives for background subagents. Mutually exclusive — see Q3.
- SDK `Options.forwardSubagentText: true` ([`sdk.d.ts:1376-1379`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L1376)) so live subagent messages include text/thinking, not just `tool_use`/`tool_result`.
- New SDK service bindings: `listSubagents` and `getSubagentMessages` on [`IClaudeAgentSdkService`](./claudeAgentSdkService.ts) / `IClaudeSdkBindings`.
- Telemetry counter `claude.subagent.resolved_via.<strategyName>` per resolution so we can observe migration when SDK ships native correlation.

**Out of scope**

- **Persistence of any subagent state.** No `claude.subagents` overlay row, no DB extension. The relationship is fully inferable from live SDK signals + on-demand scan; persistence would buy nothing on a cold path. See Q4.
- **`listSessions()` filter** — verified out as unnecessary. The SDK stores subagent transcripts under a `subagents/` subdirectory and `SessionKey.subpath` distinguishes them; `listSessions` already returns only top-level entries. See Q9.
- Multi-level (nested) subagent transcripts. The SDK keeps only one level of `<root>/subagents/agent-<id>.jsonl` — nested subagents recursively flatten under the same root. Phase 12 ships single-level navigation; nested replay is a follow-up if/when it surfaces.
- Subagent session creation persistence (DB row per child session). The host creates the child session in-memory in [`AgentSideEffects._handleSubagentStarted`](../agentSideEffects.ts#L461) on every restart by replaying the parent's transcript and emitting `subagent_started` from the live mapper; replay-only navigation works without it.
- **`IClaudeParentTranscriptResolver` interface** — premature abstraction. Parent path is one SDK call + one mapper call; wrapping it in a strategy chain for symmetry would buy nothing today. If the SDK ever forces parent-side strategy choice (e.g. paginated, remote sessionStore), upgrade `_loadParentTranscript` into its own resolver without touching consumers.
- Truncate / fork on subagent URIs — owned by [Phase 6.5](./roadmap.md).
- Subagent-specific permission UX, customizations, or model selection. Subagents inherit the parent's `Options`.
- Telemetry / OTel for subagent spans — owned by [Phase 14](./roadmap.md). The strategy-counter telemetry above is a small in-resolver counter, not full span instrumentation.

## Approach

The host already has all of the *server-side* subagent plumbing — [`AgentSideEffects`](../agentSideEffects.ts) creates child sessions silently on `subagent_started`, buffers signals that arrive before their `subagent_started` ([`agentSideEffects.ts:95-105, 308-317`](../agentSideEffects.ts#L95)), and tears down on `subagent_completed`. Phase 12 is the **Claude-side emitter** for those signals plus the Claude-side **transcript fetch** for subagent URIs.

### Empirical SDK reality (verified 2026-05-14 against live SDK and 2,525+ on-disk JSONL files)

Documented in full at [CONTEXT.md M7 §"Subagent correlation: SDK gaps and resolution strategy"](./CONTEXT.md). Headline facts that drive this plan:

1. **`SessionMessage.parent_tool_use_id` is always `null`.** Empirically verified: zero non-null occurrences across 2,525 files / 110k+ envelopes / live `getSubagentMessages` calls. The field is typed as the literal `null` (not `string | null`). Reading it on the replay path produces silently-wrong correlations. The corresponding live-stream field on `SDKAssistantMessage` IS populated — that's the live mapper's join, not a replay primitive.
2. **`agentId` and `toolUseResult.agentId` are stripped by SDK normalization.** The on-disk JSONL has both, but `getSubagentMessages` and `getSessionMessages` return only `[type, uuid, session_id, message, parent_tool_use_id, timestamp]` — agentId is gone.
3. **The SDK does inject a synthetic `agentId: <id>` text block as the LAST entry of every Agent `tool_result.content[]`.** Reliable when present; format may vary by SDK version. This is the cleanest cold-path primitive.
4. **`Agent.tool_use.input.prompt` ≡ subagent's first user message body** — byte-for-byte identical in every sampled case. Slower but bulletproof fallback.
5. **`listSubagents` returns alphabetically (`readdir`) order, NOT invocation order.** Tested across 27 sessions: 1/27 happened to match invocation order; 23 had count mismatches because the directory accumulates orphans (nested subagents, sidechain branches, compacted history). Cannot be used as a positional join.
6. **`CanUseTool` callback exposes `agentID`** for inner subagent tools. Free live correlation source.
7. **Compacted/forked sessions** have a synthetic first SDK entry; the SDK's chain-walking excludes pre-compaction Agent calls. Not a Phase 12 concern (those subagents are unreachable by design), but documented so future readers don't try to "fix" it.

There is no first-class SDK API today that maps `(parentSessionId, toolCallId) → agentId`. The resolver layer carries multiple imperfect strategies behind one stable interface; when the SDK ships native correlation, one strategy slots in and the others delete.

### Architecture — single ingress, layered strategies, no persistence

```
ClaudeAgent.getSessionMessages(uri)
        │
        ▼
IClaudeTranscriptService.getTranscript(uri)
        │
        ├── isSubagentSession(uri)? ──► IClaudeSubagentResolver.getTranscript(uri)
        │                                       │
        │                                       ├── 0. In-memory cache hit
        │                                       │      (populated by noteLiveAgentId
        │                                       │       and notePrimingFromParent)
        │                                       │
        │                                       │      ┌─ on miss: chain ─┐
        │                                       ├──────│ 1. TextSuffixStrategy   (parse agentId text block in tool_result)
        │                                       │      │ 2. PromptMatchStrategy  (compare parent prompt body to subagent first user message)
        │                                       │      │ 3. NativeStrategy       (when SDK ships it — placeholder)
        │                                       │      └──────────────────┘
        │
        └── else ──► _loadParentTranscript(uri)
                          │
                          ├── _sdkService.getSessionMessages(...)
                          ├── mapSessionMessagesToTurns(...)              ← Phase 13
                          └── _subagentResolver.notePrimingFromParent(uri, transcript)
                                                                          ← cache priming
```

Subagent path: complex inside, simple outside. Parent path: simple inside, simple outside. Both: same ingress.

### The four named strategies

| # | Strategy | When it hits | Cost | Reliability |
|---|---|---|---|---|
| — | **In-memory cache** | After first lookup OR after parent transcript load OR after canUseTool sees agentID | O(1) | 100% when populated |
| 1 | **TextSuffixStrategy** | Cold path; parse the synthetic `agentId: <id>` block from the parent's `Agent.tool_result.content[]` last entry | One regex per tool_result; reuses already-loaded parent transcript when called via `notePrimingFromParent` | High when present; missing in some sessions / SDK versions |
| 2 | **PromptMatchStrategy** | Cold path fallback when TextSuffix misses | `listSubagents` + N × `getSubagentMessages(limit:1)`, compare first user message text vs parent's `Agent.tool_use.input.prompt` | Bulletproof when prompts are non-empty; slow |
| 3 | **NativeStrategy** | TODO: when SDK ships first-class lookup | O(1) | TBD |

`LiveCaptureStrategy` is *not* in the chain — it's a write path (`noteLiveAgentId`) that feeds the in-memory cache. Conceptually it's "strategy 0" but architecturally it's a feeder, not a chain participant.

### Cache priming — why first-click-after-restart feels native

The workbench can only navigate to a subagent URI after loading the parent transcript (the marker is constructed from parent data). The transcript service exploits this dependency: after `_loadParentTranscript` succeeds, it calls `resolver.notePrimingFromParent(parentURI, transcript)`. The resolver runs `TextSuffixStrategy` over the entire transcript and bulk-populates the cache. By the time the user clicks a subagent marker, the lookup is O(1).

This means PromptMatchStrategy's slow N-fetch path only fires when:
- Parent transcript loaded but TextSuffixStrategy found no agentId text block for that toolCallId (older session / SDK version variation).
- Subagent URI is reached without loading the parent first (programmatic / test access).

Both are uncommon. The common case is a hot cache.

3. **✅ Extend `ClaudeMapperState` with subagent bookkeeping.** Add four private maps + two sets, public accessors, and explicit drain semantics:
   - `_subagentSpawningTools: Map<tool_use_id, { agentName: string; displayName: string; description?: string }>` — populated in `startToolBlock` when `toolName ∈ {'Task','Agent'}`. `agentName` / `description` come from the `tool_use.input` (`subagent_type` field for Task, declaration name for Agent — fall back to `toolName` if missing); `displayName` from [`getClaudeToolDisplayName`](./claudeToolDisplay.ts).
   - `_announcedSubagents: Set<parent_tool_use_id>` — guards `subagent_started` so it fires exactly once per subagent regardless of how many inner messages arrive.
   - `_innerToolToParent: Map<inner_tool_use_id, parent_tool_use_id>` — populated when the live mapper sees an inner `tool_use` (the message itself has envelope `parent_tool_use_id !== null`). Step 5's canUseTool agentID-bridge consumes this via a **public accessor** `getParentToolCallId(innerToolUseId: string): string | undefined` (necessary because `claudeCanUseTool.ts` lives outside the mapper and cannot read private fields). The accessor mirrors the existing public `lookupToolCall` on `ClaudeMapperState`.
   - **`_backgroundTasks: Set<tool_use_id>`** — populated when the mapper sees `SDKTaskStartedMessage` (`message.type === 'system' && message.subtype === 'task_started'`) and the message's `tool_use_id` matches an entry in `_subagentSpawningTools`. This set is the foreground/background discriminator (see step 7) and resolves the previously-open question about whether `SDKTaskStartedMessage` is needed — it is, because background `tool_result` arrives immediately and cannot be distinguished from foreground completion any other way.
   - `_completedSubagents: Set<tool_use_id>` — tracks which subagents have already fired `subagent_completed` (foreground OR background). Lives for the full session lifetime, NOT drained at turn end (a long-running background subagent's `task_notification` may arrive turns after its parent `tool_result`).
   - Drain semantics: extend `clearPendingToolCalls` (called from `mapResult`) to drop entries from `_subagentSpawningTools` only when both (a) no `task_result` has been seen AND (b) the tool_use_id is not in `_backgroundTasks` (background subagents intentionally outlive turn boundaries). Warn once per genuine orphan. Happy-path removal from `_subagentSpawningTools` happens in step 7's completion sites. `_announcedSubagents`, `_backgroundTasks`, and `_completedSubagents` are NOT drained on turn end.
   - Files: [claudeMapSessionEvents.ts](./claudeMapSessionEvents.ts).
   - Depends on: step 2 only for type imports.
   - Done when: unit tests cover all six collections' lifecycle, the public accessor, the new drain semantics, and the `task_started` ingestion path.

4. **✅ Define `IClaudeSubagentResolver` + strategy classes.** New file [`claudeSubagentResolver.ts`](./claudeSubagentResolver.ts):
   - `IClaudeSubagentResolver` interface with three methods: `getTranscript(subagentURI, token): Promise<readonly Turn[]>`, `noteLiveAgentId(parentURI, toolCallId, agentId): void`, `notePrimingFromParent(parentURI, transcript): void`.
   - `ISubagentLookupStrategy` interface with `name: string` (for telemetry) and `lookup(parent, toolCallId, ctx): Promise<string | undefined>`.
   - Three strategy classes:
     - `TextSuffixStrategy` — given a parent transcript (or fetches one if not supplied via `ctx`), scans every Agent `tool_result.content[]` for the synthetic last-text-block matching a **tolerant** regex `/^\s*agentId:\s+([a-z0-9]+)\b/im` (case-insensitive, lenient whitespace, anchored only by line start — robust to minor format drift between SDK versions). Returns the matching agentId. The same scan, when called from `notePrimingFromParent`, populates *every* `(toolCallId, agentId)` pair it finds in one pass.
     - `PromptMatchStrategy` — calls `listSubagents` for the parent, then for each agentId calls `getSubagentMessages(limit:1)` and compares the first user message text against the parent's `Agent.tool_use.input.prompt`. Match → that agentId is the answer. Validate inputs with **locally-defined** `vObj` / `vString` validators in [`claudeSubagentResolver.ts`](./claudeSubagentResolver.ts) (do NOT import from `extensions/copilot/.../claudeSessionSchema.ts` — layering violation per [coding-guidelines](../../../../../.github/instructions/coding-guidelines.instructions.md)). The validator shapes mirror the extension's pattern but live in `platform/agentHost`. No `as any` casts.
     - `NativeStrategy` — placeholder. Returns `undefined` for now. Documented TODO with the SDK feature it'll consume when shipped.
   - `ChainedSubagentResolver` — composes strategies in order, first non-undefined hit wins. Cache hits short-circuit before the chain runs. Emits telemetry counter `claude.subagent.resolved_via.<strategy.name>` (or `cache` / `none`) per resolution.
   - Cache: `Map<parentSessionId, Map<toolCallId, agentId>>` field on the resolver; cleared on parent session disposal (see step 9).
   - DI: register as a **singleton** via `createDecorator<IClaudeSubagentResolver>` in `agentHostMain.ts` / `agentHostServerMain.ts` (matches the existing Claude-service singleton pattern at `agentHostMain.ts:116-121`). Resolved in Decisions — not an open question.
   - Files: NEW `claudeSubagentResolver.ts`.
   - Depends on: step 2.
   - Done when: unit tests cover each strategy in isolation, the chain ordering, the priming side effect, the telemetry counter, AND the tolerant regex's handling of whitespace/case variations.

7. **✅ Emit `subagent_completed`.** The protocol explicitly distinguishes foreground from background subagents because parent-tool completion is unreliable for the latter ([`agentSideEffects.ts:576-581`](../agentSideEffects.ts#L576)). **Critical ordering reality** (council 2026-05-14): for background subagents, `tool_result` always arrives FIRST (the parent doesn't wait); `SDKTaskNotificationMessage` arrives later when the background work actually finishes. The discriminator is `_backgroundTasks` (populated from `SDKTaskStartedMessage`, step 3), NOT arrival order.

   Three trigger sites:

   - **Ingest `SDKTaskStartedMessage`** — add a branch in the mapper switch for `message.type === 'system' && message.subtype === 'task_started'`. If `message.tool_use_id` is in `_subagentSpawningTools`, add it to `_backgroundTasks`. This must arrive BEFORE the parent `tool_result` for the foreground/background discriminator to work (the SDK emits `task_started` before yielding the parent's `tool_result` for background mode).
   - **Foreground completion (parent `tool_result`)** — in `mapUserMessage`, when a `tool_result` lands for a `tool_use_id` in `_subagentSpawningTools`:
     - If `tool_use_id` is in `_backgroundTasks`: skip (background completion will arrive later via `task_notification`).
     - Else if `tool_use_id` is not in `_completedSubagents`: append `IAgentSubagentCompletedSignal` after the existing `SessionToolCallComplete` action, add to `_completedSubagents`, and **remove from `_subagentSpawningTools`** (happy-path cleanup so the orphan drain doesn't warn on it).
   - **Background completion (`SDKTaskNotificationMessage`)** — add a branch for `message.type === 'system' && message.subtype === 'task_notification'`. When `status ∈ {'completed','failed','stopped'}` and `tool_use_id` is set and is still in `_subagentSpawningTools` AND not in `_completedSubagents`: emit `IAgentSubagentCompletedSignal`, add to `_completedSubagents`, and remove from `_subagentSpawningTools` and `_backgroundTasks`.

   This ordering is the only correct interpretation:
   - Foreground subagent: never in `_backgroundTasks` → `tool_result` fires completion. ✅
   - Background subagent: in `_backgroundTasks` → `tool_result` is skipped, later `task_notification` fires completion. ✅
   - Background subagent where `task_notification.tool_use_id` is absent (per [`sdk.d.ts:3399`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L3399)): no completion fires. Orphan drain in `clearPendingToolCalls` does NOT touch entries in `_backgroundTasks`, so the entry survives across turns. Acceptable: the child session stays alive until the agent host shuts down. Rare edge case; document as known limitation.

   - Files: [claudeMapSessionEvents.ts](./claudeMapSessionEvents.ts).
   - Depends on: step 6.
   - Done when: unit tests cover (a) foreground tool_result fires completion, (b) background `task_started` then `tool_result` does NOT fire completion, (c) background `task_started` then `tool_result` then `task_notification` fires completion exactly once, (d) `task_notification` without prior `task_started` (defensive) treats as foreground, (e) orphan drain leaves `_backgroundTasks` entries alone.

8. **✅ Implement `IClaudeTranscriptService` + wire `ClaudeAgent.getSessionMessages` to it.** New file [`claudeTranscriptService.ts`](./claudeTranscriptService.ts):

        ```typescript
            this._logService.warn('[Claude] notePrimingFromParent failed', err);

        ```
     7. Return `turns`.
   - Resilience: every SDK call wrapped in try/catch with warn-log + return `[]` (mirror [`claudeAgent.ts:619-622`](./claudeAgent.ts#L619)). Priming has its own try/catch (above) and never propagates errors.
   - Replace [`claudeAgent.ts:611`](./claudeAgent.ts#L611) `getSessionMessages` body with a one-liner: `return this._transcriptService.getTranscript(session, CancellationToken.None);`.
   - Move the existing parent-only logic from `claudeAgent.ts` into `_loadParentTranscript`. Old code path goes away entirely.
   - Files: NEW `claudeTranscriptService.ts`, [claudeAgent.ts](./claudeAgent.ts).
   - Depends on: steps 2, 4 (resolver must exist).
   - Done when: integration test covers `getTranscript(parentURI)` (existing parent path), `getTranscript(subagentURI)` (new resolver path), nested URI walks, all SDK errors → `[]`, verifies `notePrimingFromParent` is invoked with the loaded transcript, AND verifies a deliberately-faulted priming does NOT propagate.

## Decisions

- **Q1 — Subagent-spawning tool names.** Recognise both `'Task'` and `'Agent'`. Confirmed by [`sdk.d.ts:36, 95`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts) and matched by Phase 13's [`SUBAGENT_TOOL_NAMES`](./claudeReplayMapper.ts#L137). Reuse the same constant in the live mapper and in `TextSuffixStrategy`.
- **Q2 — Where does `subagent_started` come from?** From the **live mapper**, on first sighting of a non-null envelope `parent_tool_use_id` on `SDKAssistantMessage` / `SDKUserMessage` / `SDKPartialAssistantMessage`. NOT from `SessionMessage` (the replay envelope's field is always null per Q11), NOT from a separate SDK event. Production reference at [`claudeMessageDispatch.ts:232`](../../../../../../extensions/copilot/src/extension/chatSessions/claude/common/claudeMessageDispatch.ts#L232) uses the same trigger.
- **Q3 — `subagent_completed` triggers.** Discriminated by `_backgroundTasks` (populated from `SDKTaskStartedMessage`), NOT by arrival order. For background subagents, `tool_result` always arrives FIRST (immediately, parent doesn't wait) and `task_notification` arrives LATER when the background work finishes. The plan's original "whichever fires first" mutual-exclusion was empirically wrong — see council 2026-05-14 finding #1. Foreground branch checks `_backgroundTasks` and skips if present; background branch fires on `task_notification` terminal status. `_completedSubagents` still guards against double-fire defensively. The roadmap is explicit that "background subagents continue running after the parent tool call returns" ([`agentSideEffects.ts:576-581`](../agentSideEffects.ts#L576)).
- **Q4 — Where does `agentId` come from?** *Inference, not persistence* (user steer + 4-model council unanimous). The strategy chain in `IClaudeSubagentResolver`:
  1. **In-memory cache hit** — populated by `noteLiveAgentId` (canUseTool feeder) and `notePrimingFromParent` (TextSuffix bulk-scan after every parent load).
  2. **`TextSuffixStrategy`** — parses the synthetic `agentId: <id>` block the SDK injects into every Agent `tool_result.content[]`. Reliable when present; format may shift between SDK versions.
  3. **`PromptMatchStrategy`** — `listSubagents` + N × `getSubagentMessages(limit:1)`, match first-user-message text against parent's `Agent.tool_use.input.prompt`. Bulletproof (the prompt body is verbatim shared) but slow.
  4. **`NativeStrategy`** — placeholder. Slot in at chain position 1 once the SDK ships first-class correlation.

  No DB row, no `ClaudeSessionMetadataStore` extension. The cache is process-lifetime; restart re-pays the cold path once per parent session on first navigation, but priming makes that cost amortise across subsequent clicks. See CONTEXT.md M7's "Subagent correlation" subsection for the full empirical justification.
- **Q5 — Nested subagent URIs.** Parse iteratively up to root before calling SDK; mirrors [`copilotAgent.ts:1063`](../copilot/copilotAgent.ts#L1063). The walk is genuinely required because `getSubagentMessages` only accepts the root sessionId; nested transcripts flatten on disk under the same root ([`sdk.d.ts:906`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L906)). Phase 12 ships URL-level navigation only; deeper inlining is out of scope.
- **Q6 — Persistence dropped.** Originally a `claude.subagents` overlay row on `ClaudeSessionMetadataStore`. Reverted: the relationship is fully inferable from SDK signals, persistence adds DB schema + JSON codec + race handling for marginal benefit on a cold path. The user explicitly steered this direction. If a future caller proves a sub-second cold-restart navigation requirement, revisit then.
- **Q7 — Resilience policy.** Match Phase 13's [`claudeAgent.ts:619`](./claudeAgent.ts#L619): warn + `[]` on every failure path. One bad subagent transcript must not break sibling navigation. Same rule applies inside every strategy — a strategy that throws is treated as "miss" and the chain continues.
- **Q8 — Spawning-tool lookup miss.** When `subagent_started` fires for a `parent_tool_use_id` whose `Task`/`Agent` `tool_use` was never observed (transcript truncation, mid-resume), fall back to `agentName: 'subagent'`, `displayName: 'Subagent'`, `description: undefined`. Reachable only in edge cases (truncation/compaction); the normal resume path replays the `Task` tool_use first.
- **Q9 — `listSessions` filtering dropped.** SDK file layout (`<sessionId>/subagents/agent-<id>.jsonl`, [`sdk.d.ts:906`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L906)) plus the `SessionKey.subpath` discriminator ([`sdk.d.ts:3540-3555`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L3540)) confirm `listSessions` already returns only top-level transcripts. No host-side filter needed.
- **Q10 — Reuse Phase 13's mapper for subagent transcripts.** Yes — same `SessionMessage[]` → `Turn[]` shape, same M7 grouping rules. Subagent transcripts will themselves contain `Task`/`Agent` markers in nested cases, which the mapper already flattens to subagent-marker `ToolCall`s — those navigate to deeper-nested URIs that step 8's iterative walk handles.
- **Q11 (CORRECTED) — `SessionMessage.parent_tool_use_id` is empirically always null.** The previous version of this Q claimed the value lived on the inner `.message` and was readable via cast. **Wrong.** Empirical scan of 2,525 files / 110k+ envelopes / live SDK calls: zero non-null occurrences in either location. The literal `null` type at [`sdk.d.ts:3566`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L3566) is accurate. The replay path has NO `parent_tool_use_id` join key — that's why Q4's strategy chain exists. The corresponding LIVE field on `SDKAssistantMessage` IS populated and is what step 6 reads.
- **Q12 — `forwardSubagentText: true`.** Required so live subagent stream includes text/thinking, not just `tool_use`/`tool_result` ([`sdk.d.ts:1376-1379`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L1376)). Without it, live subagent sessions are content-empty while replay returns full transcripts — a silent UX regression.
- **Q13 — `pending_confirmation` for inner subagent tools.** The mapper-derived `parentToolCallId` does NOT cover `pending_confirmation` signals because those originate from the `canUseTool` callback path, not from the message-stream mapper. Step 5 explicitly bridges `CanUseTool.agentID` ([`sdk.d.ts:187`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L187)) into the signal. Without this, inner subagent permission prompts land on the parent session and the resulting `SessionToolCallReady` action has no matching `SessionToolCallStart`.
- **Q14 — `listSubagents` is `readdir`-sorted, not invocation-ordered.** Tested against 27 real sessions: only 1 happened to match invocation order; 23 had count mismatches because the `subagents/` directory accumulates orphans (nested subagents, sidechain branches, compacted history). A coworker proposed positional matching (Nth Agent tool_use → Nth listSubagents result); rejected after empirical falsification. `listSubagents` is a discovery primitive (find all subagent files), NOT a correlation primitive. The strategy chain always pairs it with content-based matching.
- **Q15 — Compacted/forked sessions filter pre-compaction Agent calls.** Sessions continued from compaction have a synthetic first SDK entry (`message.model === '<synthetic>'`); the SDK's chain-walking excludes pre-compaction `Agent` `tool_use` blocks even though they appear in the raw JSONL. This means subagents from before the compaction boundary are unreachable via `getSessionMessages`. Not a Phase 12 concern (those subagents are unreachable by design — there's no marker to click), but documented so we don't try to "fix" it.
- **Q16 — Why `IClaudeTranscriptService` exists at all.** Without it, every consumer of subagent transcripts would branch on `isSubagentSession(uri)` and then call either the resolver or the parent-load path. Hiding this branch behind one method makes the resolver's strategy chain a pure implementation detail; it also gives the parent path the natural priming hook (every successful parent load triggers `notePrimingFromParent`). The parent path stays simple inside (no strategies of its own — see "Out of scope" rationale).
- **Q17 (NEW — council 2026-05-14) — `SDKTaskStartedMessage` is required, not optional.** The previous open question ("should we use SDKTaskStartedMessage as an earlier signal?") is resolved: yes, it's the foreground/background discriminator for `subagent_completed` (step 7, Q3). Without ingesting it, background subagents are completed prematurely when their immediate parent `tool_result` arrives. The mapper populates `_backgroundTasks` from `task_started.tool_use_id`.
- **Q18 (NEW — council 2026-05-14) — `PromptMatchStrategy` duplicate-prompt false positives are a known limitation, not "correct."** Previous plan claimed the deterministic-first-match behavior was correct. Reviewers (Codex, Opus) correctly pointed out that deterministic ordering doesn't equal correlation: two Agent invocations with identical prompt bodies (e.g. retried delegation, parallel identical sub-tasks) cannot be distinguished by PromptMatch alone. Mitigation: TextSuffixStrategy runs first and resolves the common case unambiguously (each tool_result has its own toolCallId+agentId pair in the suffix). PromptMatch fires only when TextSuffix missed for that toolCallId. Documented as an inherent strategy limitation in Risks; the Native strategy (when SDK ships it) closes the gap.
- **Q19 (NEW — council 2026-05-14) — DI scope of `IClaudeSubagentResolver`: singleton.** Resolved (was open). Existing Claude services in the agent host are process-singletons ([`agentHostMain.ts:116-121`](../agentHostMain.ts#L116), [`agentHostServerMain.ts:208-216`](../agentHostServerMain.ts#L208)) and there's one Claude provider registration per process. Singleton matches the existing pattern; `disposeParent(parentURI)` from `ClaudeAgent.disposeSession` (step 9) clears the relevant slice. Per-`ClaudeAgent`-instance was considered and rejected as inconsistent.
- **Q20 (NEW — council 2026-05-14) — `claudeCanUseTool.ts` requires expanded DI seam.** The current `IClaudeCanUseToolDeps` only takes `getSession` + `configurationService` ([`claudeCanUseTool.ts:20`](./claudeCanUseTool.ts#L20)). Step 5 expands it with `getMapperState` and `subagentResolver`. `ClaudeMapperState` also gains a public `getParentToolCallId(innerToolUseId): string | undefined` accessor (private field reads from outside the module are not allowed); mirrors the existing public `lookupToolCall` accessor.

## Risks

- **PromptMatchStrategy cannot disambiguate identical prompt bodies.** If two Agent invocations in the same parent session use the *same* `tool_use.input.prompt` (e.g. retried delegation, parallel identical sub-tasks), PromptMatch cannot tell which toolCallId maps to which agentId. Mitigation: TextSuffixStrategy runs first and resolves the common case unambiguously — PromptMatch fires only as a fallback when TextSuffix missed for that specific toolCallId. The cache + live capture also typically cover this case. **Inherent strategy limitation**, not a bug; the `NativeStrategy` (when SDK ships) closes the gap entirely. Documented in Q18.
- **Scan cost on first navigation of a parent with N subagents (cache miss path)** — `listSubagents` + up to N × `getSubagentMessages(limit:1)` for `PromptMatchStrategy`. Mitigated by (a) `TextSuffixStrategy` running first and usually hitting on every Agent tool_result without any extra SDK calls, (b) `notePrimingFromParent` populating the cache opportunistically when the parent transcript loads, (c) process-lifetime cache persistence. The slow chain-fallthrough fires only when both TextSuffix and the cache miss — uncommon.
- **`SDKTaskNotificationMessage.tool_use_id` is optional** ([`sdk.d.ts:3399`](../../../../../../node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts#L3399)). When absent on a background subagent's terminal notification, completion cannot fire and the spawning-tool entry survives across turns. Mitigated by the `_backgroundTasks` orphan-skip in `clearPendingToolCalls` (step 3): the entry doesn't generate a warning, but the child session stays alive until process shutdown. Rare edge case; documented as a known limitation.
- **TextSuffixStrategy depends on the SDK's synthetic text-block format being stable.** Format observed today: `agentId: <hex> (use SendMessage with to: '<hex>') ...`. The tolerant regex (`/^\s*agentId:\s+([a-z0-9]+)\b/im`, see step 4) handles minor whitespace/case drift but cannot survive structural changes (e.g. removal of the suffix entirely). If Anthropic changes the wording, TextSuffix silently misses; PromptMatchStrategy picks up the slack and telemetry tells us via the `resolved_via.prompt_match` counter rising. This is exactly why the chain has multiple strategies — no single primitive is bulletproof.
- **`SDKTaskStartedMessage` ordering vs parent `tool_result`.** Step 7 requires `task_started` to arrive before the parent `tool_result` so `_backgroundTasks` is populated when `tool_result` is evaluated. Empirically the SDK emits `task_started` first; documented but not contractually guaranteed. Defensive interpretation: if `tool_result` arrives for a tool_use_id not in `_backgroundTasks`, treat as foreground (current behavior). A later `task_notification` for the same id would no-op via `_completedSubagents`. Worst case: a background subagent is incorrectly treated as foreground and its child session is torn down early. Mitigation: integration test asserts the actual ordering against a stubbed SDK that emits `task_started` first.
- **Live capture only works for subagents whose tools require canUseTool prompts.** A read-only subagent (e.g. only invokes `Read`/`Bash` with auto-approval) may never trigger `canUseTool`, so the live mapping isn't built — first navigation falls back to the chain. Acceptable; the chain handles it.
- **`AgentSideEffects` buffering provides defense in depth.** Mapper prepend ordering of `subagent_started` is the primary mechanism; the buffer at [`agentSideEffects.ts:307-317`](../agentSideEffects.ts#L307) catches any future case where the mapper splits emissions. Documented invariant: every batch with a `parent_tool_use_id`-bearing signal includes its `subagent_started` (when not yet announced) at index 0.
- **Subagent restoration on host restart already works in the workbench renderer** because Phase 13 emits subagent **markers** (`_meta.toolKind = 'subagent'` + `ToolResultContentType.Subagent`). Phase 12 only adds the *fetch* path; live child sessions are in-memory only.
- **Mapper state lifecycle on yield-restart.** The `ClaudeMapperState` lives at the [`ClaudeSdkMessageRouter`](./claudeSdkMessageRouter.ts#L33) level and survives the [`ClaudeSdkPipeline`](./claudeSdkPipeline.ts) `rebind` path ([line 345, 386](./claudeSdkPipeline.ts#L345)). The new subagent collections (`_subagentSpawningTools`, `_announcedSubagents`, `_innerToolToParent`, `_backgroundTasks`, `_completedSubagents`) follow the same convention as `_toolCallTurnIds`: cross-message state survives queue rebinds intentionally, since the SDK replays from `resume: sessionId` and the same subagent state should still apply. Re-emission of `subagent_started` is guarded by `_announcedSubagents`.
- **Strategy class explosion.** Three strategies + chain + interface + service is a lot of new files for a feature that could nominally be done inline. Justification: each strategy has a distinct failure mode (TextSuffix can silently miss, PromptMatch has identical-prompt false positives, Native doesn't exist yet) and the migration story (delete two strategies when SDK ships native) is genuinely cleaner with named classes than an inline switch. If review pushes back, collapse `ChainedSubagentResolver` + the strategies into one file — the interface separation is the load-bearing part.

### Step 10 — Architecture deepening: resolver owns subagent state (landed)

After the council review, an architecture pass identified that the `mapperState` accessor chain (Router → Pipeline → Session) existed only so the canUseTool bridge could read one piece of subagent correlation state (`getParentToolCallId`). The fix moves all five Phase-12 collections (`spawning`, `background`, `innerToParent`, `announced`, `completed`) off `ClaudeMapperState` and onto `IClaudeSubagentResolver`, keyed by parent session URI via a per-parent `IParentSubagentTracker`. The resolver becomes the single domain authority for subagent correlation: cache (read+write), live tracker (read+write), strategy chain, transcript fetch, parent disposal.

Concretely:
- `IClaudeSubagentResolver` gains 9 verbs: `noteSpawningTool`, `getSpawningTool`, `clearSpawningTool`, `noteBackgroundTask`, `isBackgroundTask`, `noteInnerToolEdge`, `getParentToolCallId`, `markAnnounced`, `markCompleted`, plus a turn-end drain `drainOrphanedSpawningTools`.
- `ClaudeMapperState` shrinks back to its pre-Phase-12 size (10 verbs); the file's `SUBAGENT_SPAWNING_TOOL_NAMES` constant stays as the mapper still classifies tool names at `content_block_start`.
- `mapSDKMessageToAgentSignals` takes a new `subagentResolver: IClaudeSubagentResolver` arg; every state.X subagent call becomes `resolver.X(session, ...)`. `tagWithParent` keeps `state` as a second arg for the spawning-tool name lookup (which legitimately lives in `_toolCallNames`).
- `ClaudeSdkMessageRouter` injects `IClaudeSubagentResolver` and threads it through. The `mapperState` accessor on Router / Pipeline / Session is **deleted entirely**.
- `claudeCanUseTool.ts`'s `resolveSubagentParent` calls `deps.subagentResolver.getParentToolCallId(session.sessionUri, options.toolUseID)` directly — no more Session→Pipeline→Router walk.
- `disposeParent` evicts both the cache and the live tracker.

### Step 12 — Critical fix: mapper drops inner subagent content (landed)

Manual launch on 2026-05-14 surfaced a regression: subagent Task tool calls render in the parent transcript as "Running agents" with **no inner content** and never complete. Agent host log proved it:

- 5 inner `assistant` + 4 inner `user` messages with non-null `parent_tool_use_id`
- **0** `stream_event` partials with non-null `parent_tool_use_id`
- 2 `tool_result for unknown tool_use_id` warnings from `mapUserMessage`

**Root cause**: the Phase 12 plan assumed `forwardSubagentText: true` would deliver inner content via `SDKPartialAssistantMessage` (`stream_event`). Empirically the SDK delivers inner content **only** as canonical `SDKAssistantMessage` / `SDKUserMessage` envelopes. `mapAssistantCanonical()` returned `[]` for every inner message, so:

- inner text / thinking blocks: never emitted (no `SessionResponsePart`)
- inner `tool_use` blocks: never registered (no `state.startToolBlock` to populate `_toolCallTurnIds`)
- inner `tool_result`: hit the unknown-tool-use-id warn path because the lookup misses

A second corollary was found in the same investigation: **the SDK does NOT invoke the parent's `canUseTool` callback for inner subagent tools** (it runs them in `bypassPermissions` internally). So the Phase 12 step 5 bridge — wired correctly — never fires for inner tools. The only signal source for inner tool calls is the canonical assistant message.

**Fix**: rewrote `mapAssistantCanonical` to handle inner messages (`parent_tool_use_id !== null`) by walking `message.content` and emitting one signal per block:

- `text` / `thinking` → `SessionResponsePart` (Markdown / Reasoning) with the full content
- `tool_use` → `SessionToolCallStart` + `SessionToolCallReady` (with `confirmed: NotNeeded` to auto-transition through PendingConfirmation since inner tools skip the host gate) + register the `tool_use_id` via `state.startToolBlock` so the matching `tool_result` can pair back, AND `subagentResolver.noteInnerToolEdge(...)` so any future canUseTool with `agentID` (rare top-level case) still has the parent edge available

`tagWithParent` then tags every emitted action with `parent_tool_use_id` and `AgentSideEffects` routes them to the subagent session.

Top-level (`parent_tool_use_id === null`) behaviour is unchanged: partials still cover everything, the canonical message still returns `[]`.

CONTEXT.md M7 updated with the empirical correction and the canUseTool-skip finding. New regression test in [claudeMapSessionEvents.test.ts](../../test/node/claudeMapSessionEvents.test.ts) drives an inner canonical assistant + inner tool_result, asserts text + tool-call signals fire and complete with `parentToolCallId` set on every action.

Validation: 138 unit tests green (was 137; +1 regression). Full type-check + layer-check clean. Awaiting user manual re-test.

The Agent tool's input carries `subagent_type` (e.g. `"Explore"`, `"general-purpose"`) and `description` which would render much better. Plumbing those requires extending `noteSpawningTool` to carry metadata and writing it from the canonical `assistant` (where `tool_use.input` is complete) rather than `content_block_start` (where input is empty).

PromptMatch should pick up the slack at replay time. Worth re-instrumenting after a few real sessions to see the actual `resolved_via.<name>` distribution.

### 2026-05-14 (empirical SDK probing + design rewrite)

Investigations after the initial review changed several conclusions:

- **`SessionMessage.parent_tool_use_id` is empirically always `null`.** Scanned 2,525 files / 110k+ envelopes / live `getSubagentMessages` calls — zero non-null occurrences in either envelope or inner message. The literal `null` type is accurate. Q11 corrected; the previous "read from .message" guidance was wrong.
- **The SDK strips every correlation field on replay.** `agentId` (subagent envelope), `toolUseResult.agentId` (parent envelope), `parent_tool_use_id` (everywhere) — all gone from `getSessionMessages` / `getSubagentMessages` output.
- **The SDK injects a synthetic `agentId: <id>` text block in Agent `tool_result.content[]`.** Reliable when present (3/3 in tested session), missing in some sessions/versions. This is the cleanest cold-path primitive.
- **Prompt body is byte-for-byte identical** between parent's `Agent.tool_use.input.prompt` and subagent's first user message. Bulletproof fallback when text-suffix misses.
- **`listSubagents` returns alphabetically (`readdir`) order, NOT invocation order.** Tested across 27 sessions — only 1/27 happened to match invocation order; 23 had count mismatches due to orphaned subagent files. Cannot be used for positional matching.
- **Plan rewritten:** introduced `IClaudeTranscriptService` (single ingress) + `IClaudeSubagentResolver` (strategy chain) + `notePrimingFromParent` (cache pre-population from parent loads). Step 7's "scan via `parent_tool_use_id`" replaced entirely. New steps 4 (resolver) and 8 (transcript service). Q11 corrected, Q14 (listSubagents ordering) and Q15 (compaction) and Q16 (transcript service rationale) added.
- **All findings captured in [CONTEXT.md M7 §"Subagent correlation"](./CONTEXT.md)** for future sessions.

### 2026-05-14 second pass (4-model council review of the rewrite)

After the rewrite, council reviewed and surfaced more issues. All addressed:

1. **BLOCKING (3/3 reviewers) — Background-subagent completion fires too early.** The original mutual-exclusion logic assumed "whichever fires first wins," but `tool_result` for background subagents ALWAYS arrives first (the parent doesn't wait). The foreground branch would always win and prematurely tear down the still-running subagent. **Fix:** ingest `SDKTaskStartedMessage` into a new `_backgroundTasks: Set<tool_use_id>`; the foreground branch checks this set and skips when present, so only the later `task_notification` fires completion. Resolves the previously-open question about `SDKTaskStartedMessage` (Q17). Step 3, step 7, Q3, and Risks all updated.
2. **BLOCKING (3/3 reviewers) — `canUseTool` bridge has no path to mapper state.** Step 5 originally said `state._innerToolToParent[toolUseID]`, but `ClaudeMapperState` is private to a different module and `claudeCanUseTool.ts` has narrow deps. **Fix:** expanded `IClaudeCanUseToolDeps` with `getMapperState` and `subagentResolver`; added a public `getParentToolCallId(innerToolUseId)` accessor on `ClaudeMapperState` (mirrors the existing `lookupToolCall` accessor); extended `requestPermission` / `requestUserInput` args with `parentToolCallId?`. Step 5 and Q20 rewritten.
3. **BLOCKING (2/3 reviewers) — PromptMatch "deterministic first match is correct" is wrong.** Deterministic readdir ordering doesn't equal correct correlation. Two identical prompt bodies can resolve to the wrong agentId. **Fix:** acknowledged honestly as an inherent strategy limitation in Risks and Q18; mitigation is that TextSuffixStrategy runs first (and the synthetic suffix is per-tool_use_id, so it doesn't have this issue), and the future Native strategy closes it entirely.

## Step 14 — `SubagentRegistry` refactor + E2E validation

After the live wiring landed, an architecture sweep collapsed the
junk-drawer `IClaudeSubagentResolver` singleton (13-method DI service
keyed by `parentUri` everywhere) into a per-session domain model:

- **`SubagentSpawn`** — one record per Task `tool_use`, owns its own
  lifecycle state: `agentId` (set-once via `setAgentId`), `background`
  (mutable bool flipped on `task_started`), `subagentType` /
  `description` (metadata from the `tool_use.input` written at
  canonical-assistant time), and idempotency latches `markAnnounced` /
  `markCompleted`.
- **`SubagentRegistry`** — per-`ClaudeAgentSession` collection of
  `SubagentSpawn` + reverse index from inner `tool_use_id` → parent
  Task. Owned by the session (via `_register`), dies with it
  automatically — no `disposeParent` needed, no `parentUri` parameter
  on any method.
- **Free functions** in `claudeSubagentResolver.ts`: the strategy
  chain (`TextSuffixStrategy` → `PromptMatchStrategy` → `NativeStrategy`),
  `resolveAgentIdViaChain` (pure orchestrator), `getSubagentTranscript`
  (the only public entrypoint for subagent transcript fetch). The
  parent-transcript-fetch + registry-priming logic was inlined directly
  into `ClaudeAgent.getSessionMessages` (~25 lines, no separate file).
- **`canUseTool` bridge** reads `session.subagents.getParentSpawn(...)`
  directly; the agent dropped its `IClaudeSubagentResolver` injection
  and the deps passed to `handleCanUseTool`.
- **DI cleanup** — both `IClaudeSubagentResolver` and the short-lived
  `IClaudeTranscriptService` decorator are gone from
  `agentHostMain.ts` / `agentHostServerMain.ts`.

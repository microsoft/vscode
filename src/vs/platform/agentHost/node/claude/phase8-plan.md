# Phase 8 — File Edit Tracking

**Status: done** (unit-tested + live E2E verified).

> Generated by super-planner. Source: `roadmap.md` (phase 8).
> Last updated: 2026-05-08 after council synthesis (GPT / Opus / Gemini) + grill pass.

**Status:** ready

## Goal

Expose file reads/writes/edits as `ToolResultFileEditContent` in `SessionToolCallComplete` signals so the workbench can render before/after diffs and support per-file accept/reject undo. Mirrors the behavior of `FileEditTracker` in the Copilot path.

This phase also promotes `FileEditTracker` (and its session-db URI helpers) out of `node/copilot/` into `node/shared/`, fixing pre-existing cross-folder imports in `agentService.ts` and `sessionDiffAggregator.ts`.

## Scope

**In scope:**
- Move `FileEditTracker`, `buildSessionDbUri`, `parseSessionDbUri`, `ISessionDbUriFields` from `node/copilot/fileEditTracker.ts` → `node/shared/fileEditTracker.ts` and update all import sites.
- Wire `ClaudeFileEditObserver` to the SDK message stream in `ClaudeAgentSession` so assistant tool-use events call `trackEditStart(filePath)` and matching user tool-result events call `completeEdit`/`takeCompletedEdit` and stage content on mapper state.
- Add `isClaudeFileEditTool(toolName)` + `getClaudeEditFilePath(toolName, input)` helpers to `claudeToolDisplay.ts` (covers Write, Edit, MultiEdit, NotebookEdit; NOT Bash, TodoWrite).
- Add a per-session `FileEditTracker` owned by `ClaudeAgentSession`.
- Extend `ClaudeMapperState` with `cacheFileEdit(toolUseId, content)` + `takeFileEdit(toolUseId)` to pre-stage `ToolResultFileEditContent` before the synchronous mapper fires.
- Update `mapUserMessage` to append cached file edits to `SessionToolCallComplete.result.content`.
- Set `enableFileCheckpointing: true` in `Options` (safe — doesn't affect DB-backed undo path, enables future `rewindFiles` validation).
- Unit tests in `claudeAgent.test.ts` + `claudeMapSessionEvents.test.ts`. Integration test (proxy-backed). Smoke row in `smoke.md`.

**Out of scope:**
- `Query.rewindFiles()` as the production undo path (granularity is wrong for per-file selective undo — see Decisions §D3). A validation unit test confirms the API doesn't break anything.
- Bash-tool edit tracking (documented gap — `sed -i`, `cat > file` aren't surfaced as structured file-edit tool events in the SDK message stream).
- Abort/steering (Phase 9), client tools (Phase 10), customizations (Phase 11), subagents (Phase 12), transcript reconstruction (Phase 13).

## Prerequisites

- Phase 7 DONE: `SessionToolCallComplete` already emitted, `ClaudeMapperState` carries cross-message `tool_use_id` → `turnId`/`toolName` maps, `canUseTool` wired.
- `FileEditTracker`, `buildSessionDbUri`, `parseSessionDbUri` exist at `node/copilot/fileEditTracker.ts`.
- `ISessionDataService.openDatabase(sessionUri)` returns a ref-counted `IReference<ISessionDatabase>` (Copilot pattern at `copilotAgentSession.ts:264`).
- `IInstantiationService` is needed to construct `FileEditTracker` with DI-injected `IFileService`, `IDiffComputeService`, `ILogService`.

## Approach

Wire `ClaudeFileEditObserver` to the existing SDK message stream boundary. On assistant messages with file-edit tool uses (`Write`, `Edit`, `MultiEdit`, `NotebookEdit`), the observer snapshots before-state via `FileEditTracker.trackEditStart(filePath)` and records `tool_use_id → path`. On subsequent user messages with matching `tool_result` entries, it calls `completeEdit` + `takeCompletedEdit` and caches `ToolResultFileEditContent` on `ClaudeMapperState` keyed by `tool_use_id`. `mapUserMessage` remains synchronous and appends whatever staged file-edit content is available for that tool call.

The `FileEditTracker` is moved to `node/shared/` (not copied) because `agentService.ts` and `sessionDiffAggregator.ts` already import it from the Copilot folder — fixing these cross-folder imports is low-risk scope that removes an existing layering smell.

Per-file undo is DB-backed: the workbench reads before/after content via `resourceRead` on `session-db:` URIs, then writes the before-content back to disk to reject an edit. The SDK's `rewindFiles` is not used in the production path (wrong granularity for selective per-file undo).

```ts
/**
 * Returns true for tools that produce on-disk file edits tracked by
 * FileEditTracker. Excludes TodoWrite (in-memory todo list, no
 * filesystem write) and Bash (edits not surfaced to SDK hooks).
 */
export function isClaudeFileEditTool(toolName: string): boolean {
    return toolName === 'Write' || toolName === 'Edit'
        || toolName === 'MultiEdit' || toolName === 'NotebookEdit';
}

/**
 * Extract the edited file path from the tool input for file-edit tools.
 * Returns undefined for non-file-edit tools or malformed input.
 */
export function getClaudeEditFilePath(
    toolName: string,
    input: unknown,
): string | undefined {
    if (typeof input !== 'object' || input === null) { return undefined; }
    const rec = input as Record<string, unknown>;
    switch (toolName) {
        case 'Write':
        case 'Edit':
        case 'MultiEdit': {
            const fp = rec.file_path;
            return typeof fp === 'string' ? fp : undefined;
        }
        case 'NotebookEdit': {
            const fp = rec.notebook_path;
            return typeof fp === 'string' ? fp : undefined;
        }
    }
    return undefined;
}
```

### Step 3 — Extend `ClaudeMapperState` with file-edit cache ✓

**Files:** `node/claude/claudeMapSessionEvents.ts`, `test/node/claudeMapSessionEvents.test.ts`

**Change:** Add a cross-message `_completedFileEdits` map to `ClaudeMapperState`. The map is populated by the `PostToolUse` hook (Step 5) and consumed by `mapUserMessage`.

```ts
// In ClaudeMapperState:
private readonly _completedFileEdits = new Map<string, ToolResultFileEditContent>();

/**
 * Called from the PostToolUse hook handler after `takeCompletedEdit`
 * resolves. Stored here so the synchronous mapper can read it when
 * the tool_result user message arrives.
 *
 * The SDK guarantees the hook completes (and the subprocess receives
 * the hook response) before emitting the tool_result user message, so
 * this map is always populated by the time mapUserMessage fires.
 */
cacheFileEdit(toolUseId: string, content: ToolResultFileEditContent): void {
    this._completedFileEdits.set(toolUseId, content);
}

/**
 * Consume and remove the cached file edit for this tool_use_id.
 * Returns undefined if no edit was tracked (e.g. the tool was denied
 * or was not a file-edit tool).
 */
takeFileEdit(toolUseId: string): ToolResultFileEditContent | undefined {
    const content = this._completedFileEdits.get(toolUseId);
    if (content) { this._completedFileEdits.delete(toolUseId); }
    return content;
}
```

Hook handler methods (called from `Options.hooks`):
```ts
/**
 * Phase 8: maps tool_use_id → filePath for edit tools. Populated in
 * handlePreToolUse; consumed and removed in handlePostToolUse. Entries
 * for denied tools (PostToolUse never fires) are dropped on session dispose.
 */
private readonly _editToolPaths = new Map<string, string>();

/**
 * Called from Options.hooks.PreToolUse. Snapshots the file's current
 * content as the "before" state for edit tools.
 *
 * Note: PreToolUse fires BEFORE canUseTool / permission check. If the
 * user denies the tool, PostToolUse will NOT fire, leaving a dangling
 * _editToolPaths entry and a _pendingEdits entry in the tracker.
 * These are bounded by session lifetime and cause no corruption.
 */
async handlePreToolUse(
    toolName: string,
    toolInput: unknown,
    toolUseId: string,
): Promise<void> {
    if (!isClaudeFileEditTool(toolName)) { return; }
    const filePath = getClaudeEditFilePath(toolName, toolInput);
    if (!filePath) { return; }
    this._editToolPaths.set(toolUseId, filePath);
    try {
        await this._editTracker.trackEditStart(filePath);
    } catch (err) {
        this._logService.warn(`[ClaudeAgentSession] PreToolUse trackEditStart failed: ${err}`);
        this._editToolPaths.delete(toolUseId);
    }
}

/**
 * Called from Options.hooks.PostToolUse. Captures the "after" state,
 * computes the diff, stores to DB, and caches the ToolResultFileEditContent
 * on the mapper state so mapUserMessage can append it to
 * SessionToolCallComplete synchronously.
 *
 * The subprocess is blocked waiting for this hook's return value before
 * emitting the tool_result SDKMessage, so the cache is always populated
 * before mapUserMessage fires.
 */
async handlePostToolUse(toolUseId: string, currentTurnId: string): Promise<void> {
    const filePath = this._editToolPaths.get(toolUseId);
    if (!filePath) { return; }  // non-edit tool or denied before PreToolUse ran
    this._editToolPaths.delete(toolUseId);
    try {
        await this._editTracker.completeEdit(filePath);
        const fileEdit = await this._editTracker.takeCompletedEdit(
            currentTurnId, toolUseId, filePath,
        );
        if (fileEdit) {
            this._mapperState.cacheFileEdit(toolUseId, fileEdit);
        }
    } catch (err) {
        this._logService.warn(`[ClaudeAgentSession] PostToolUse edit tracking failed: ${err}`);
    }
}
```

> **Turn id in PostToolUse**: `handlePostToolUse` needs the current turn id to pass to `takeCompletedEdit`. `ClaudeAgentSession` has `_inFlightRequests[0]?.turnId` which is the turn id of the in-flight request — valid because the single-in-flight invariant is maintained by `_sessionSequencer`.

## Decisions

Resolved during council synthesis + grill pass. All five decisions are binding for implementation.

- **D1 — FileEditTracker location: move to `node/shared/`** — `agentService.ts` and `sessionDiffAggregator.ts` already import from `node/copilot/fileEditTracker.ts`; these are non-Copilot files making a cross-folder import (confirmed via `grep`). Moving to `node/shared/` is four import-path updates and fixes the existing layering smell. NOT copied into `node/claude/` (would duplicate 200 lines of identical logic) and NOT left in `node/copilot/` (cross-agent import would propagate the smell). GPT initially disagreed (calling it import churn), but this import already exists outside the copilot path — moving is net-negative churn.

- **D2 — Undo mechanism: DB snapshots only, not `rewindFiles`** — `Query.rewindFiles(userMessageId)` rewinds ALL tracked files at once. Selective per-file undo requires either `rewindFiles` + re-apply accepted files, or direct `IFileService.write(beforeContent)`. The DB-backed snapshot path (already implemented in `FileEditTracker`) provides before/after content as `session-db:` URIs; the workbench reads them via `resourceRead` and writes the before-content to disk on reject. This is the same mechanism the Copilot path uses. `rewindFiles` is deferred to a follow-up if the team wants bulk-rewind support.

- **D3 — `enableFileCheckpointing: true` in Options** — Set it now as forward-compatible prep. It doesn't affect the DB-backed undo path. A unit test validates the SDK's `rewindFiles` API shape doesn't throw (`dryRun: true`) so future phases can adopt it without guessing. If enabling it causes test failures (subprocess overhead, unexpected file mutations), flip to `false` and document.

- **D4 — Cache on `ClaudeMapperState`, not a callback into the session** — The mapper is currently synchronous. Pre-computing `ToolResultFileEditContent` in the async `PostToolUse` hook and caching on `ClaudeMapperState` keeps the mapper synchronous, avoids threading async callbacks through the existing `mapSDKMessageToAgentSignals` contract, and follows the same "cross-message state lives in mapper state" pattern established by Phase 7's `toolCallTurnIds`.

- **D5 — `TodoWrite` excluded from file edit tracking** — `TodoWrite` updates Claude's in-memory todo list (no filesystem write). Including it would produce misleading diff output for a non-file artifact. `getClaudeEditFilePath('TodoWrite', input)` returns `undefined` by design.

## Risks

- **`PostToolUse` hook timing invariant** — The plan relies on the SDK subprocess being blocked on the hook response (`Promise<HookJSONOutput>`) before it emits `tool_result`. The `AsyncHookJSONOutput = { async: true }` variant allows non-blocking hooks; our hooks return `SyncHookJSONOutput = {}` so the subprocess waits for full resolution. Validate in the integration test by asserting `ToolResultFileEditContent` is non-empty on `SessionToolCallComplete`. If the invariant fails (race condition observed in production), fall back to making `mapUserMessage` async with a small per-tool-call `await` fence.

- **`PreToolUse` fires before permission — dangling pre-edit entries** — `PreToolUse` fires before `canUseTool`. If the user denies the tool, `PostToolUse` does not fire. This leaves a dangling `_editToolPaths` entry and a `_pendingEdits` entry in `FileEditTracker` (containing the file's before-content as a `VSBuffer` in memory). These entries are bounded by the session's lifetime and disposed with the session tracker. No corruption occurs: `completeEdit` checks `_pendingEdits.get(filePath)` before doing anything; `takeCompletedEdit` checks `_completedEdits.get(filePath)`. Mitigation already in the design: `_editToolPaths.delete(toolUseId)` in the `handlePreToolUse` error path prevents the file-path mapping from being stale.

- **`IInstantiationService` DI ordering** — Adding `@IInstantiationService` to `ClaudeAgent`'s constructor requires it to be declared BEFORE the non-service params (there are none in the current constructor, so ordering is not a concern). See Phase 1.5 learning: `GetLeadingNonServiceArgs` strips from the end, so branded service params must come last.

- **DB ref lifecycle** — `ClaudeAgentSession` must register `dbRef` via `this._register(dbRef)` (not `dbRef.object`) to ensure the `IReference` is disposed when the session is disposed. If the session disposes the db ref but the agent host tries to re-use the same session URI for another session, the DB will reopen cleanly (refcounted).

- **`FakeQuery.rewindFiles` in tests** — The `FakeQuery` in `claudeAgent.test.ts` doesn't currently implement `rewindFiles`. Add a stub that returns `{ canRewind: false }` so Test H can pass without a real SDK subprocess.

- **Bash-tool edits not tracked** — `sed -i file`, `cat > file`, and similar shell-mediated edits are not surfaced to `PreToolUse`/`PostToolUse` hooks because the SDK hooks fire per-tool-call, not per filesystem event. This is a documented gap in `roadmap.md`. Consider a file-watcher diff fallback in a follow-up phase.

5. In the workbench, open the diff view for the edited file. Confirm before/after content renders.
6. Reject the edit (click "Undo" or equivalent); confirm the file reverts to pre-edit state.
7. Repeat with a second edit and accept; confirm the file retains the edit.

Run `verify-claude-logs.sh --phase=8` after the operator session to assert the IPC log contains the required signal shape.

### Manual

If E2E tooling is unavailable, run the dev build (`scripts/code.sh`) manually and perform the edit → diff render → accept/reject cycle from the console.

### Notable design choices

- `_completedFileEdits` is keyed only by `tool_use_id` (not `(turnId, toolUseId)`) because the SDK guarantees `tool_use_id` uniqueness within a session and Phase 6's per-session sequencer keeps at most one in-flight turn at a time.
- The `PostToolUse` hook resolves `{}` (a `SyncHookJSONOutput`) on every path including errors, so a tracker failure (e.g. `IFileService` read error) is logged but never blocks the SDK from emitting the matching `tool_result`. The user still sees the tool result; only the file-edit content block is missing.
- `handlePreToolUse` writes to `_editToolPaths` BEFORE awaiting `trackEditStart`, so a denial that fires between Pre and Post hooks (which leaves a dangling map entry plus a `_pendingEdits` `VSBuffer` in the tracker) is bounded by session lifetime — both maps are disposed when the session disposes.
- `dbRef` is registered in the `ClaudeAgentSession` constructor BEFORE the `WarmQuery` abort registration so the database stays open while the SDK's in-flight `tool_result` is being processed; the order is enforced by the `_register` chain in `Disposable.dispose`.

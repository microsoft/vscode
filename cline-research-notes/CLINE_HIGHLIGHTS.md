# Cline — highlights from this codebase

This file was generated by scanning the **local** Cline repository. It summarizes capabilities that stand out in the code and official docs, with emphasis on **using models without a classic third-party API key** (OAuth, subscription-backed CLIs, IDE integrations).

**Method:** Five incremental passes over disjoint areas of the tree; each pass appended or refined sections below.

---

### Pass 1 — evals, CI, activation, programmatic API

- **Testing pyramid** documented in `evals/ARCHITECTURE.md`: (1) deterministic contract/unit tests, (2) **smoke tests** (`evals/smoke-tests/`) with curated scenarios (create file, edit, multi-file, TypeScript function, apply-patch, Gemini/OpenAI-compat variants), **pass@k** / flakiness metrics in `evals/analysis/`, (3) **Harbor / cline-bench** E2E agent runs (`evals/e2e/`).
- **CI workflows** (`.github/workflows/`): main `test.yml`, `e2e.yml`, **`cline-evals-regression.yml`**, CLI TUI tests, **nightly** npm/SDK publish pipelines, pack-cli, JetBrains-triggered flows, stale handling.
- **VS Code activation** (`src/extension.ts`): migrates legacy VS Code storage, **exports native storage to shared file-backed `~/.cline/data/`** so VS Code / CLI / JetBrains share state; registers sidebar webview, commands, **HookDiscoveryCache** file watchers; syncs auth on secret changes **across windows** (`onDidChange` on `cline:clineAccountId`).
- **Public VS Code extension API** (`src/exports/`): `createClineAPI()` exposes `startNewTask`, `sendMessage`, `pressPrimaryButton`, `pressSecondaryButton` for automation/tests (`src/test/cline-api.test.ts`).
- **Standalone / headless context** (`src/standalone/vscode-context.ts`): minimal `ExtensionContext` shim with `CLINE_DIR`, `INSTALL_DIR`, `WORKSPACE_STORAGE_DIR` env vars — used when the same core runs outside full VS Code.

### Pass 2 — webview ↔ core bridge, git UX, worktrees

- **Protobuf/gRPC-style IPC, not ad-hoc postMessage**: the webview sends `grpc_request` / `grpc_request_cancel` payloads (`WebviewMessage`); the host dispatches through `handleUnaryRequest` and `handleStreamingRequest` in `src/core/controller/grpc-handler.ts`, correlating `request_id` for cancellation and streaming chunks.
- **Proto surface area** (`proto/cline/`): dedicated services for **tasks**, **state** (incl. onboarding), **checkpoints**, **MCP**, **browser**, **files**, **account**, **hooks**, **slash** commands, **web**, **UI**, **worktrees**, plus `proto/host/` for window/workspace/diff/testing — this is how JetBrains/CLI can share the same contract as VS Code.
- **Git commit messages from diffs** (`src/hosts/vscode/commit-message-generator.ts`): command `cline.generateGitCommitMessage` uses **staged-first** diff (`getGitDiffStagedFirst`), optional developer notes, and prompts for **conventional commits**; uses `buildApiHandler` so the **same provider picker** drives commit generation.
- **Worktree workflow** (`proto/cline/worktree.proto`, controller handlers): list/create/delete/switch worktrees, `.worktreeinclude` helpers, branch checkout, **merge back + optional delete** — geared toward parallel agent branches without trashing the main working tree.
- **Rules & project conventions**: workspace layout uses **`.clinerules/`** for workflows and hook scripts; tools like `readDirectory` can **exclude** `.clinerules/workflows` and `.clinerules/hooks` from bulk listing to avoid noise (`src/utils/fs.test.ts` documents intended behavior).

### Pass 3 — remote policy, focus chain, onboarding, UX scheduling

- **Remote config / org controls** (`state-keys.ts`): server-driven fields include `remoteConfiguredProviders`, **remote rules/workflow/skills toggles**, **remote global rules/workflows/skills** lists — supports enterprises that predefine allowed providers and shared prompts (see `openFile.ts` routing for rule vs workflow vs skill URIs).
- **Focus chain** (`FocusChainSettings`, `focus-chain-utils.ts`, system prompt `write_to_file.ts`): optional **task progress checklists** in tool outputs; `/smol` and `/compact` slash commands and **deep-planning** take `focusChainSettings` into account.
- **Yolo mode** (`yoloModeToggled` in global state): when on, plan-mode wording **drops** some clarifying-question guidance; tool handlers reference **yolo-mode mistake limits** (e.g. `ListFilesToolHandler`) — a stricter auto-pilot path with guardrails.
- **Onboarding model catalog** (`src/shared/cline/onboarding.ts`): curated **FREE** starter models with scores/latency badges; can be **overridden by feature-flag payload** for A/B or regional rollout.
- **Task presentation scheduler** (`TaskPresentationScheduler.ts`): debounces/prioritizes UI flushes (`immediate` vs `normal`) so streaming tool output stays responsive without starving the event loop.
- **Multi-root workspaces**: `WorkspaceRoot` entries carry **VCS type** (Git/Mercurial/none) and optional **commit hash** (`src/shared/multi-root/types.ts`); `primaryRootIndex` drives default cwd for hooks and operations when several folders are open.

### Pass 4 — networking, environments, observability

- **Unified `fetch`** (`src/shared/net.ts`): all providers should pass this `fetch` into SDKs — **critical for JetBrains + CLI** where `undici` + `EnvHttpProxyAgent` applies `HTTP(S)_PROXY`; VS Code relies on its own proxy-aware global fetch. Documents **corp TLS** (`NODE_EXTRA_CA_CERTS`), SOCKS/PAC gaps, and `~/.cline/cline-core-service.log` for triage.
- **Multi-environment endpoints** (`src/config.ts`): **production**, **staging**, **local** (`localhost:3000` / `7777`), and **self-hosted / on-prem** via endpoints file — swaps `appBaseUrl`, `apiBaseUrl`, `mcpBaseUrl`.
- **Self-hosted safety rail**: uninitialized config is treated as **self-hosted** so telemetry/feature code does not accidentally phone home (`ClineEndpoint.shouldDisableExternalCalls` pattern); PostHog and **build-time OTEL to Cline** are skipped in `Environment.selfHosted` (`TelemetryProviderFactory`).
- **Telemetry options**: **PostHog** and/or **OpenTelemetry** (`TelemetryProviderFactory`, `remoteConfigToOtelConfig`) — OTEL can **bypass user opt-out** when env/org mandates (`bypassUserSettings`).

### Pass 5 — notebooks, i18n, auto-approval, editor affordances

- **Jupyter / notebook path**: dedicated commands (`jupyterGenerateCell`, `jupyterExplainCell`, `jupyterImproveCell` in `registry.ts`); **`NotebookDiffView`** and `sanitizeNotebookForLLM` strip noise before LLM context (`integrations/misc/notebook-utils.ts`, `DiffViewProvider`).
- **Localization**: translated README and **`locales/`** trees for several languages (community-contributed per README banner links).
- **Granular auto-approval** (`AutoApprovalSettings`): per-action toggles for **reads inside/outside workspace**, **edits**, **safe vs all terminal commands**, **browser**, **MCP** — not just a single global switch.
- **`.clineignore`**: blocked paths increment `consecutiveMistakeCount` (yolo guardrail) and return structured errors instead of leaking ignored content (`ListFilesToolHandler`, `clineIgnoreController` references throughout tools).
- **Misc registry commands** worth knowing: **add terminal output to chat**, **fix / explain / improve** selection, **reconstruct task history**, **worktrees** button, **walkthrough** opener — all in `src/registry.ts` with VS Code registration in `extension.ts`.

## Using Claude, ChatGPT / Codex, and similar — without an API key

There is **no single switch** that routes *your existing Anthropic console API key* or *OpenAI platform key* “as if” it were a consumer subscription. Instead, Cline supports several **parallel auth paths**:

| Path | What you sign into | API key? | Notes (from code/docs) |
|------|-------------------|----------|------------------------|
| **Cline provider** | `app.cline.bot` (Google, GitHub, or email) | No | Billing is through Cline credits; models are served via Cline’s backend (`ClineHandler` → OpenAI-compatible client against `ClineEnv.config().apiBaseUrl`). This is **not** the same as wiring your personal Anthropic/OpenAI *subscription* directly—it’s Cline-hosted access. |
| **Claude Code** | Anthropic **Claude Code** CLI (`claude`), with **Claude Pro / Max** | No (CLI auth) | Cline **shells out** to the Claude Code CLI and streams structured output (`runClaudeCode` / `ClaudeCodeHandler`). Your subscription usage follows Anthropic’s Claude Code rules; Cline still does tools/edits itself. See `docs/provider-config/claude-code.mdx`. |
| **ChatGPT Subscription** (`openai-codex`) | OpenAI OAuth (`auth.openai.com`), **ChatGPT Plus/Pro** | No | Dedicated OAuth client config in `src/integrations/openai-codex/oauth.ts` (callback on `localhost:1455`, tokens in secret storage). UI copy: sign in for GPT-5-family models **without** a platform API key (`OpenAiCodexProvider.tsx`). |
| **GitHub Copilot** (`vscode-lm`) | VS Code **Language Model API** / Copilot | No separate Cline key | Uses VS Code’s exposed chat models (`VSCodeLmProvider` polls `ModelsServiceClient.getVsCodeLmModels`). Requires Copilot (or compatible) entitlement in the editor. |
| **Qwen Code** | Qwen OAuth (`chat.qwen.ai`, credentials under `~/.qwen`) | No | OAuth2 flow and cached creds (`qwen-code.ts`); similar “CLI / ecosystem auth” idea. |
| **Oracle Code Assist (OCA)** | OCA OAuth / tokens via `OcaAuthService` | No | Enterprise Oracle path; not a general consumer subscription. |

**Plain English:**

- **“I have Claude Pro/Max and don’t want an Anthropic API key”** → install and authenticate **Claude Code**, then choose provider **Claude Code** in Cline and point to your `claude` binary.
- **“I have ChatGPT Plus/Pro and don’t want an OpenAI API key”** → choose **ChatGPT Subscription** / OpenAI Codex provider and complete the **Sign in to OpenAI Codex** OAuth flow.
- **“I don’t want to manage any provider keys at all”** → use the **Cline** provider and sign in at **app.cline.bot** (Cline’s own billing and model access—not raw Anthropic/OpenAI billing).
- **“I already pay for Copilot in VS Code”** → try **GitHub Copilot** / `vscode-lm` and pick an available language model.

**BYOK** (bring your own key) remains the usual **Anthropic**, **OpenAI**, **OpenRouter**, **Bedrock**, **Vertex**, etc.—documented in `docs/getting-started/authorizing-with-cline.mdx`.

## “Really interesting” product / engineering features

### Agent loop and safety

- **Human-in-the-loop tools**: file writes, terminal commands, browser actions, MCP calls, etc., go through approval flows; aligns with README positioning (not a fire-untrusted autonomous runner inside your repo).
- **Plan vs act modes**: separate behaviors with `plan_mode_respond` in plan mode vs full tool access in act mode (`Controller.togglePlanActMode`, system prompt variants under `src/core/prompts/system-prompt/`).
- **Dual mode API config**: optional different models/providers for plan and act (`getModeSpecificFields`, mode-specific VS Code LM selectors).

### Editor & workspace integration

- **Terminal execution** with shell integration (long-running commands, “proceed while running”, streamed output).
- **Diff-based edits** with linter/diagnostic feedback loop so the model can repair mistakes.
- **Checkpoints**: git-backed snapshots to compare/restore workspace (and task) state (`src/integrations/checkpoints/`).
- **Timeline / history**: changes integrate with VS Code file history (as described in README).

### Context and cost awareness

- **Context manager** tracks edits, truncations, and metadata so history can be rolled back coherently with checkpoints (`ContextManager`).
- **@ mentions**: `@file`, `@folder`, `@url`, `@problems`, etc., to steer context without burning turns on implicit searches.
- **Usage / cost tracking** for API tasks (README; implemented in billing/telemetry paths).

### Browser automation

- **Browser tool** for navigate/click/type/scroll/screenshots/console (`BrowserToolHandler`) — supports interactive debugging and E2E-style flows tied to local dev servers.

### MCP (Model Context Protocol)

- **MCP servers, tools, prompts, resources** — including OAuth-gated servers documented as needing *upfront* tokens in config (`loadMcpDocumentation.ts` guidance).
- Tooling to **load MCP docs** and **use MCP tools** from the agent loop.

### Extensibility

- **Hooks** (similar in spirit to Cursor hooks): `TaskStart`, `TaskResume`, `TaskCancel`, `TaskComplete`, `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Notification`, `PreCompact` — discovered from workspace/global hook dirs, with combined runners and streaming (`hook-factory.ts`, `HookDiscoveryCache`, `executeHook`).
- **Slash commands** embedded in user text: `newtask`, `smol`/`compact`, `newrule`, `reportbug`, `deep-planning`, `explain-changes`, plus MCP prompt commands like `/mcp:server:prompt` (`parseSlashCommands`).
- **Skills** surface in tooling (`UseSkillToolHandler`) aligned with “skills” workflows in rules/context.
- **Subagents** (`UseSubagentsToolHandler`, `SubagentRunner`) for delegated work.
- **System prompt registry** (`PromptRegistry`, tags/variants per provider and model family) — see `src/core/prompts/system-prompt/README.md` for contribution patterns.

### Providers & transports (breadth)

The dropdown list in `src/shared/providers/providers.json` includes **Cline**, **ChatGPT Subscription**, **Gemini**, **Anthropic**, **Bedrock**, **Copilot**, **OpenAI** / **OpenAI-compatible**, **Claude Code**, **Ollama**, **LM Studio**, **OpenRouter**, gateways (**Vercel AI Gateway**, **LiteLLM**), and many hosted inference vendors (**Groq**, **Cerebras**, **Fireworks**, **Together**, **Baseten**, etc.).

Implementation highlights:

- **Retries**, stream normalization, and **tool-call processors** shared across providers (`src/core/api/`).
- **OpenAI Codex** uses **Responses / WebSocket** modes where applicable (`OpenAiCodexHandler`).
- **Azure** OpenAI supports **DefaultAzureCredential** bearer path (`openai.ts`).

### CLI & ACP

- **`cline` CLI** with auth that understands Cline account + Codex OAuth credentials (`cli/src/utils/auth.ts`).
- **ClineAgent** implements an ACP-style agent: session modes, OAuth for `cline-oauth` and `openai-codex-oauth` (`cli/src/agent/ClineAgent.ts`).
- Config **import** from competitor CLIs (e.g. Codex `auth.json`) — `cli/src/utils/import-configs.ts` (API keys only where applicable; OAuth types handled separately).

### Telemetry & feature flags

- **PostHog** (product analytics + flags) and **OpenTelemetry** export — see **Pass 4** for self-hosted / bypass behavior (`TelemetryProviderFactory`, `PostHogFeatureFlagsProvider`).

## Practical limitations called out in docs (Claude Code)

From `docs/provider-config/claude-code.mdx`:

- Images become placeholders.
- No prompt caching on that path.
- Responses are not always fine-grained streamed char-by-char (CLI batching behavior).
